# Подсистема памяти Horae

Horae — система памяти/контекста. Она совмещает две идеи SillyTavern:

- **World Info** — записи срабатывают, когда в недавнем тексте встречаются их
  **ключевые слова** (`keywords`).
- **Снимки состояния** — записи с `always_on=True` подмешиваются **всегда** (инвентарь,
  скрытые характеристики, текущее положение сюжета, описание сцены).

Плюс третья, автоматическая: **авто-сводка сюжета** (см. ниже) — фоновая задача,
которая сама сжимает прошедшие события в always_on-запись, чтобы история,
вылетевшая из окна контекста, не терялась.

Код: `backend/horae_memory.py`. Модель: `HoraeEntry` (см. [DATA_MODEL.md](DATA_MODEL.md)).

## Области видимости записи

Запись Horae может быть привязана так:

| `session_id` | `character_id` | Смысл |
|:---:|:---:|---|
| NULL | NULL | **Глобальный лор мира** — виден во всех чатах (и всем пользователям) |
| задан | — | **Память конкретного чата** (приватная для владельца чата) |
| — | задан | **Лорбук персонажа** — применяется в чатах с этим персонажем |

В UI у каждой записи показывается тег области (🌐 глоб. / 💬 чат / 🎭 перс.).

## Как собирается контекст

`assemble_context(...)` — **чистая** функция (без БД и сети, поэтому легко покрывается
юнит-тестами `tests/test_horae_memory.py`). Порядок:

1. **Триггеры.** По тексту текущего сообщения + хвоста истории ищутся записи, чьи
   `keywords` встретились (`_scan_text_for_triggers`). `always_on`-записи добавляются
   безусловно.
2. **Системный промпт (начало)** = паспорт персонажа (`system_prompt/description/
   personality/scenario/mes_example`) + блок персоны пользователя + блок сработавших
   записей Horae (по `priority`, КРОМЕ авто-сводки) + `BEHAVIOR_GUIDE` (роль, работа
   с файлами, анти-выдумки) + `STYLE_GUIDE` (портативная разметка).
2b. **Переинъекция в КОНЕЦ** (сильнейшая позиция — recency bias). Чтобы характер не
   «плыл» и память реально влияла в длинном окне, прямо перед репликой пользователя
   добавляются системные блоки: «Что было в истории» (авто-сводка Horae), манифест
   приложенных файлов, инструкция веб-поиска (если включён), время пользователя,
   Author's Note, **якорь характера** (`_render_char_anchor`) и, в самом конце,
   **Post-History Instructions** персонажа (SillyTavern «jailbreak»/UJB).
3. **Бюджет токенов.** Несжимаемая часть — системный промпт + текущее сообщение.
   История добавляется с конца (свежие важнее), пока хватает окна контекста:
   `params.context_tokens` из UI (Генерация → «Окно контекста») либо
   `CONTEXT_TOKEN_BUDGET` из .env (по умолчанию 1 000 000 — максимум Gemini).
   Вложения истории (фото/аудио/видео) пересылаются модели заново: по умолчанию
   ВСЕ (полная память), опционально — до N МБ на ход (`history_files_mb` из UI
   «Файлы в памяти диалога» / `HISTORY_FILES_MB` из .env; не влезшие заменяются
   пометкой `[видео: имя]`). Данные достаются точечно из blob-таблицы.
4. **Author's Note** добавляется отдельным system-сообщением в самый конец контекста —
   так у заметки максимальное влияние на ответ.
5. **Аватары** (если включено `send_avatars`) — внешность персонажа и пользователя
   подмешиваются как картинки.

`build_context_from_db(...)` — обёртка: читает персонажа, персону, историю и записи
Horae (свои + лорбук персонажа, с учётом доступа) и зовёт `assemble_context`.

## Приватность

`GET /api/horae` и операции над записями скоупятся по доступу (`_can_access_horae` в
`main.py`): глобальный лор виден всем; приватная память чата и лорбук персонажа — только
владельцу, админу и соавтору расшаренного чата. Регрессия закреплена тестом
`tests/test_horae_privacy.py`.

## Откуда берутся записи

- Вручную в UI (вкладка «Память Horae»): заголовок, содержимое, ключевые слова,
  категория, `always_on`, область (глобально / только этот чат).
- При **импорте чата** (нативном или SillyTavern) — снимок состояния и хронология
  событий (см. [IMPORT_EXPORT.md](IMPORT_EXPORT.md)).
- При **импорте карточки персонажа** — лорбук (`character_book`) → записи персонажа.


## Авто-сводка сюжета (авто-память)

Каждые ~12 новых сообщений после завершения хода запускается фоновая задача
(`backend/main.py::_maybe_update_summary`): она берёт СТАРУЮ сводку + новые
события и просит модель слить их в единый конспект (до ~1500 символов).
Результат сохраняется как запись Horae этого чата:

- категория `summary`, заголовок «📜 Сводка сюжета (авто)», `always_on=True`,
  `priority=50` — подмешивается в каждый запрос;
- в `keywords` хранится служебная метка `last:<id>` — id последнего учтённого
  сообщения (чтобы не суммировать одно и то же дважды) и `__auto__`;
- запись видна во вкладке «Память» — её можно править руками или удалить
  (после удаления сводка начнётся заново);
- выключатель: вкладка «Память» → галка «Авто-сводка сюжета» (хранится в
  `/settings/ui`, ключ `auto_summary`).

Так решается «дыра в памяти»: окно контекста ограничено, но суть событий за его
пределами модель всё равно видит через сводку.
