# Канвас (рабочее пространство документ/код)

Канвас — отдельное рабочее пространство рядом с чатом для работы с большими текстами
и кодом (по мотивам Canvas в Gemini). Вместо «простыни» в узком чате материал
открывается **справа**, как отдельная сущность, которую можно править вручную и
дорабатывать с ИИ.

Код: модель `Canvas` (`models.py`), эндпоинты `/api/canvas*` (`main.py`), экспорт в
`document_service.py`, UI — панель `.canvas-pane` и методы `*Canvas*` в `app.js`.

> **Главное:** Канвас ОТВЯЗАН от сообщения пользователя. Сообщение юзера НИКОГДА не
> превращается в документ. Канвас — просмотрщик/редактор того, что СГЕНЕРИРОВАЛ ИИ.

## 1. Двухоконный режим (side-by-side)

Слева — чат, справа — Канвас (НЕ модалка поверх). При открытии к сетке `.app-grid`
добавляется класс `with-canvas` (колонка `1fr` под канвас рядом с чатом). На узких
экранах — вкладки (чат/канвас).

## 2. Поток создания (плашка-документ)

1. Пользователь жмёт в композере `[+]` → «📄 Создать документ/код (Canvas)» — это
   триггер (`canvasGenMode`), рядом с обычной отправкой.
2. Пишет запрос («Напиши большую статью») и отправляет. Текст уходит в чат как
   ОБЫЧНОЕ сообщение.
3. Фронт зовёт `POST /api/sessions/{id}/canvas_generate` → бэкенд генерирует ответ
   (нестримингово, `complete`), создаёт `Canvas` и кладёт в чат **«плашку документа»**
   — assistant-сообщение с `canvas_id` (а НЕ полотно текста). content плашки = заголовок.
4. В чате — **обычное сообщение ИИ** (интро-текст) И рядом карточка `.doc-card`
   (💻/📄 + заголовок канваса). Текст промпта пользователя в документ НЕ превращается.
5. Клик по плашке → `openCanvas(canvas_id)` (`GET /api/canvas/{id}`) → справа выезжает
   Канвас, **сразу в режиме результата** (предпросмотр).

Тип определяется по ответу: целиком код-блок ```` ``` ```` → `code`, иначе `document`.

## 2a. Редактор / Просмотр (умный предпросмотр) + единый тулбар

Переключатель `.canvas-tabs` (data `canvasView` = `edit|preview`) с единым тулбаром
слева (контекст зависит от типа канваса):
- **Веб-код / React** (`canvasIsWeb`) → «▶ Превью»: live-рендер в `<iframe srcdoc sandbox>`
  через computed `previewSrcdoc`. Тулбар: кнопка «⧉ Скопировать код» (`copyCanvas`).
- **Документ** → «👁 Просмотр»: рендер Markdown (`renderMd`). Тулбар форматирования:
  **B** / *I* / H / `</>` — оборачивают выделение Markdown-разметкой (`wrapSelection`).
- **Прочий код** → редактор + просмотр «как есть» (`<pre>`) + «Скопировать код».

`canvasIsWeb` детектит не только HTML (`<!doctype>`, `<html>`, `<body>`, `<div>+<script>`),
но и **React/JSX** (`import react`, `reactdom`, `export default` + JSX-тег).

`previewSrcdoc` собирает HTML для iframe:
- Полный HTML-документ (`<!doctype>`/`<html>`) — отдаётся как есть.
- **React/JSX** — оборачивается в страницу с CDN React 18 + ReactDOM + **Babel Standalone**
  (`type="text/babel"`), `import`/`export default` вырезаются, компонент `App`/дефолтный
  рендерится в `#root` (как Artifacts у Claude/Gemini — компилируется и запускается в браузере).
- Прочий фрагмент/CSS/JS — оборачивается в простую страницу.

`openCanvas` открывает сразу в `preview` для веб-кода и документов (показывает
результат), в `edit` — для прочего кода. Правки (ИИ или пользователь в редакторе)
реактивно отражаются в предпросмотре; канвас остаётся привязан к своей плашке-сообщению
(`source_message_id`).

## 2b. Управление состоянием: мутация vs создание (Intent Routing)

Канвас — **умный редактор**, а не генератор бесконечных файлов. Фронт помнит активный
документ (`this.canvas` с `id` — это `activeDocumentId`) и при контекстных запросах
**мутирует его**, а не плодит новые плашки. Маршрутизация в `send()`:

1. Триггер «📄 Документ» (`canvasGenMode`): если Канвас уже открыт и запрос **не**
   «с нуля» → правим открытый (`editOpenCanvas`); иначе создаём новый (`canvasGenerate`).
2. Канвас **открыт**, запрос обычным сообщением:
   - `_isNewCanvasIntent` («новый/с нуля/заново/ещё один/создай новый…») → `canvasGenerate`
     (новая плашка — пользователь явно просит новый файл).
   - `_isEditIntent` («исправь/измени/добавь/сделай длиннее/перепиши/fix/edit…») →
     `editOpenCanvas` (PATCH текущего канваса, **без** новой плашки).
   - иначе → обычное сообщение в чат (Канвас не трогаем).

`editOpenCanvas(prompt)` шлёт `POST /api/sessions/{id}/canvas_edit` с `canvas_id` +
`prompt`; бэкенд отправляет в модель **текущее содержимое + запрос**, патчит тот же
канвас (история для undo сохраняется), в чат кладёт запрос юзера и подтверждение
«✏️ Обновил …» (assistant-сообщение **без** `canvas_id`). После правки фронт
автоматически переключает `canvasView` в `preview` — результат виден сразу.

## 3. Контекстное (inline) редактирование

Ключевая фича: выделите в редакторе **фрагмент** текста или строки кода и дайте команду
(«сделай дерзче», «найди баг здесь») — ИИ перепишет **только выделенное**, остальной
документ не трогая.

Реализация: редактор — `<textarea>`; выделение ловится по `selectionStart/End`
(`captureSel` → `canvasSel`). При наличии выделения кнопка меняется на «✨ К
выделенному», а в запрос `revise` уходят `selection_start`/`selection_end`. Бэкенд
переписывает только `content[start:end]` (модель получает весь документ для контекста +
сам фрагмент) и склеивает результат обратно. Без выделения правится весь материал.

## 4. Быстрые действия — плавающий тулбар (как в Notion/Medium)

Статичных кнопок над редактором НЕТ. Тулбар всплывает у выделения, только когда
выделен фрагмент (мышью/тапом), и применяет пресет-инструкцию `revise` к этому
фрагменту. Под капотом — `onEditorPointerUp`/`onEditorKeyUp` ставят `toolbarPos` по
координатам курсора; `@mousedown`/`@scroll` его прячут; кнопки зависят от типа:
- **Текст:** Короче · Подробнее · Строже · Проще · Грамматика.
- **Код:** Комментарии · Найти баги · Ревью · Перевести на другой язык (спрашивает язык).
- **✨ Своя команда** — фокусирует нижнее поле, выделение сохраняется (правка уйдёт
  во фрагмент).

Для правки ВСЕГО материала выделение снимают и пишут команду в нижнее поле ввода.

## 5. Версионирование (undo)

Перед каждой правкой ИИ текущее содержимое кладётся в историю (`Canvas.history`, JSON,
последние 30 версий, `_push_history`). Кнопка ↩ (`/api/canvas/{id}/undo`) откатывает к
предыдущей версии. В ответе канваса есть флаг `can_undo`.

## 6. Экспорт

Кнопки 📄 (Docx) и 📑 (PDF) — `GET /api/canvas/{id}/export?fmt=docx|pdf`
(`document_service.markdown_to_docx` / `markdown_to_pdf`; PDF с кириллицей через
системный шрифт, либо точнее через LibreOffice). Имя файла — RFC 5987 (кириллица).

## Эндпоинты

| Метод | Путь | Назначение |
|------|------|-----------|
| POST | `/api/sessions/{id}/canvas_generate` | Сгенерировать НОВЫЙ документ/код → плашка в чате |
| POST | `/api/sessions/{id}/canvas_edit` | Правка ОТКРЫТОГО канваса (мутация, без новой плашки) |
| GET | `/api/canvas?session_id=` | Канвасы сессии |
| POST | `/api/canvas` | Создать (дедуп по `source_message_id`) |
| PATCH | `/api/canvas/{id}` | Ручная правка (title/kind/language/content) |
| POST | `/api/canvas/{id}/revise` | Доработка ИИ; опц. `selection_start/end` — только фрагмент |
| POST | `/api/canvas/{id}/undo` | Откат к предыдущей версии |
| DELETE | `/api/canvas/{id}` | Удалить |
| GET | `/api/canvas/{id}/export?fmt=docx\|pdf` | Экспорт |

Доступ ко всем — через сессию канваса (`_canvas_or_403`).

## Возможные улучшения

Стриминг ответа `revise` (сейчас ждём целиком), подсветка синтаксиса для кода
(textarea её не даёт — нужен оверлей или редактор вроде CodeMirror), список всех
канвасов сессии в UI.
