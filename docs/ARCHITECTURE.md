# Архитектура

## Принцип: тонкий клиент, толстый сервер

Браузер и Telegram — это просто «окна» в один и тот же бэкенд. Вся тяжёлая работа
(вызовы LLM, сборка контекста, файлы, аудио, память) происходит на сервере. Клиент
не ходит в LiteLLM напрямую и не хранит ключей.

```
┌─────────────┐   REST (fetch)      ┌──────────────────────────────────────┐
│  Браузер    │ ──────────────────► │                FastAPI                │
│  (Vue 3)    │   WebSocket/SSE     │            backend/main.py            │
└─────────────┘ ◄────────────────── │                                      │
                                    │  middleware: BasicAuth → Access →     │
┌─────────────┐   aiogram polling   │              CORS → NoCacheStatic     │
│  Telegram   │ ──────────────────► │                                      │
│  (бот)      │   telegram_runtime  │  ┌────────────┐   ┌───────────────┐   │
└─────────────┘                     │  │ horae_     │   │ llm_gateway   │   │
                                    │  │ memory     │──►│ (LiteLLM)     │──┐│
                                    │  └────────────┘   └───────────────┘  ││
                                    │  ┌────────────┐                      ││
                                    │  │ generation │  фоновые asyncio-    ││
                                    │  │ manager    │  задачи + SSE-буфер  ││
                                    │  └────────────┘                      ││
                                    └───────────────────┬──────────────────┘│
                                                        │ SQLAlchemy async   │
                                                  ┌─────▼─────┐         ┌────▼────────┐
                                                  │  SQLite   │         │ LiteLLM     │
                                                  │ aichat.db │         │ proxy :4000 │
                                                  └───────────┘         └─────────────┘
```

## Слои

1. **Транспорт / API** — `backend/main.py`. FastAPI-приложение: REST-эндпоинты,
   WebSocket `/ws/chat/{id}` для стриминга, SSE `/sse/job/{id}` для дослушивания при
   обрыве, раздача статики `frontend/` через StaticFiles. Здесь же три middleware
   (см. [SECURITY.md](SECURITY.md)).
2. **Доменная логика** — отдельные модули: сборка контекста (`horae_memory`), шлюз к
   LLM (`llm_gateway`), управление генерациями (`generation`), группы (`group_chat`),
   импорт/экспорт (`chat_import`, `native_io`, `characters`), документы
   (`document_service`), аккаунты (`accounts`), настройки (`admin_service`,
   `settings_service`).
3. **Данные** — `models.py` (ORM) + `database.py` (движок, сессии, авто-миграции).
   Одна БД на всё; см. [DATA_MODEL.md](DATA_MODEL.md).
4. **Клиенты** — `frontend/` (Vue без сборки) и `telegram_runtime.py` (aiogram).

## Жизненный цикл генерации (веб)

1. Браузер открывает `WebSocket /ws/chat/{session_id}` (с `?token=`/`?code=` для авторизации).
2. Пользователь отправляет `{"type":"user_message", content, attachments, params, reply_to_message_id}`.
3. Сервер сохраняет сообщение пользователя, собирает контекст через
   `horae_memory.build_context_from_db()` (паспорт персонажа + персона + Author's Note +
   сработавшие записи Horae + история под бюджет токенов + мультимодальный контент).
4. `generation_manager.start_runner()` запускает **фоновую asyncio-задачу**, которая
   стримит токены из `llm_gateway.stream_completion()` и шлёт их в WebSocket
   (`{"type":"token"}`), а также пишет их в буфер задачи.
5. По завершении ответ сохраняется в БД (как активный свайп), клиент получает `done`
   и перечитывает историю (сервер — источник истины).
6. **Если соединение оборвалось** — клиент дослушивает ту же задачу по
   `EventSource /sse/job/{job_id}` (буфер уже накоплен в `generation`), ответ не теряется.
7. **Стоп** (`{"type":"stop"}`) отменяет задачу; уже сгенерированный текст сохраняется.

Групповой чат идёт тем же путём, но `group_chat` решает, кто из персонажей отвечает
(по упоминанию имени или через ИИ-режиссёра), и стримит несколько реплик подряд
(`{"type":"speaker"}` / `token` / `speaker_done`).

## Жизненный цикл сообщения (Telegram)

`telegram_runtime` принимает апдейт → проверяет доступ (`_gate`) → находит/создаёт
сессию → скачивает вложения (`_attachment_from_message`) → собирает альбом в один ход
(`_handle_media`) → `build_context_from_db` → `stream_completion` (накопление в строку) →
`send_long` режет ответ и шлёт частями как Telegram-HTML. См. [TELEGRAM.md](TELEGRAM.md).

## Ключевые проектные решения

- **LiteLLM-прокси по умолчанию.** Провайдеров и ключи настраивает прокси, а приложение
  шлёт `litellm_proxy/<model>` + `api_base`. Даже для прокси без авторизации
  передаётся фиктивный ключ `DUMMY_PROXY_KEY` (иначе LiteLLM падает с AuthenticationError).
- **Обработка всегда на сервере.** Браузер не знает адреса прокси и не хранит ключей.
- **Сервер — источник истины.** После генерации клиент перечитывает историю из БД,
  а не доверяет своему локальному состоянию.
- **Одна БД на веб и бота.** Привязка Telegram к аккаунту (`User.telegram_id`) даёт боту
  доступ к приватным персонажам/чатам владельца.
- **Без сборки на фронте.** Vue с CDN + один `app.js` — нет Node, нет шага компиляции.
- **Опциональность.** Аккаунты, код доступа, Basic Auth, Telegram — всё выключаемо;
  по умолчанию приложение открыто и работает «из коробки».
